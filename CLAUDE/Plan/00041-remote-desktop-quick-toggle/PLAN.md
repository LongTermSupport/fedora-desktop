# Plan 00041: Remote Desktop Quick Toggle

**Status**: 🔄 In Progress
**Type**: Feature Implementation
**Created**: 2026-05-20
**Owner**: joseph
**Priority**: Medium

## Overview

Add a one-click GNOME quick-settings toggle to share the existing Wayland desktop
session over RDP, scoped to LAN clients only, with no persistence across reboot.
The user wants to control one laptop from another over the local network without
having to keep a remote-desktop service permanently running.

The toggle wraps `gnome-remote-desktop` in Desktop Sharing mode (existing-session
mirror), a non-`--permanent` firewalld rich-rule for LAN CIDRs, and a small GNOME
Shell extension that calls a helper CLI.

## Goals

- Single quick-settings toggle: ON enables RDP + LAN firewall hole; OFF closes both.
- Off by default. Off after every reboot. The user must deliberately turn it on.
- Firewall rule scoped to a user-configured LAN CIDR allow-list (no internet exposure, no defaults).
- Existing-session sharing — the remote viewer sees what is on the laptop's screen.
- All system changes deployed via Ansible (no manual file copies).

## Non-Goals

- Internet exposure / port-forwarding (LAN only).
- Headless / Remote Login mode (separate GDM session) — explicitly out of scope per user.
- NoMachine / AnyDesk integration — user reports instability in prior usage.
- PAM-integrated RDP credentials (impossible without falling back to xrdp/X11).
- Auto-blanking the local screen while remote sees the real desktop — not achievable
  in mirror-mode RDP (see Decision 3 for the privacy workaround).

## Context & Background

Fedora 43 ships GNOME 47, which provides `gnome-remote-desktop` as the
Wayland-native RDP server. It has two modes:

1. **Remote Login** — headless, a new login screen, separate session. Rejected.
2. **Desktop Sharing** — RDP mirror of the running session. Chosen.

Desktop Sharing has one inherent property the user must accept: locking the
local desktop also locks the remote view, because both see the same compositor.
This plan documents the workaround (Decision 3) but does NOT auto-apply it.

## Architecture

```
extensions/remote-desktop-toggle@fedora-desktop/
  ├── metadata.json
  └── extension.js                          # quick-settings toggle, shells out to rdt

files/home/.local/bin/rdt                   # toggle CLI: on / off / toggle / status
files/etc/rdt/rdt.conf                      # ALLOWED_CIDRS allow-list (overridable per host)

playbooks/imports/optional/common/play-remote-desktop-toggle.yml
                                            # installs gnome-remote-desktop, deploys
                                            # script + config + extension, enables ext
```

Runtime flow on toggle ON:

1. Extension calls `~/.local/bin/rdt on`.
2. `rdt` starts `gnome-remote-desktop.service` (user scope).
3. `rdt` adds a firewalld rich-rule per CIDR in `/etc/rdt/rdt.conf` (no `--permanent`,
   so it disappears on reboot).
4. Extension shows OK notification with listen IP.

Toggle OFF reverses the above. Reboot also reverses it (service is not user-enabled;
firewall rule is not persistent).

## Decisions

### Decision 1: Engine = gnome-remote-desktop (Desktop Sharing)

**Context**: Need Wayland-native existing-session sharing.
**Options Considered**:

1. gnome-remote-desktop Desktop Sharing — chosen.
2. xrdp — spins a separate X11 session, breaks GNOME-on-Wayland features.
3. NoMachine / AnyDesk — user reports prior instability.
4. RustDesk / WayVNC — viable but still mirror-mode (same lock-screen behaviour
   as gnome-remote-desktop), and add a third-party dependency for no functional gain.

**Decision**: gnome-remote-desktop is built-in, actively developed by Red Hat,
Wayland-native, and OCI-quality-of-life-wise the cleanest path on F43.

### Decision 2: RDP credentials live in GNOME Keyring, not PAM

**Context**: User asked whether the RDP user/pass could just be the system login.
**Reality**: `gnome-remote-desktop` does not authenticate via PAM. It stores its
own credentials in GNOME Keyring. There is no clean way to make them track the
Linux login password without replacing the engine with xrdp.
**Decision**: User sets RDP credentials once via Settings → System → Remote Desktop
UI. They never enter the repo. They are NOT auto-generated by the playbook.
This keeps secrets out of the public repo.

### Decision 3: Privacy via lid-close, not via local lock

**Context**: User wanted "local screen locked, remote still sees desktop". Not
possible in mirror-mode RDP — locking locally locks the remote view too.
**Workaround** (documented, NOT auto-configured):

- `gsettings set org.gnome.settings-daemon.plugins.power lid-close-ac-action 'nothing'`
- `gsettings set org.gnome.settings-daemon.plugins.power lid-close-battery-action 'nothing'`
- Optionally `HandleLidSwitch=ignore` in `/etc/systemd/logind.conf` for firmware-level lid.

The user can apply these from Settings → Power if they want lid-close privacy.
The playbook does not change power policy.
**Decision**: Document workaround, leave it as a manual opt-in.

### Decision 4: Firewall = non-persistent rich-rule, explicit per-host CIDR

**Context**: User said "LAN only, not on by default, not persistent".
**Decision**:

- No default — `ALLOWED_CIDRS` ships empty in `/etc/rdt/rdt.conf` and the
  script aborts with a helpful message until the user sets it. This is
  deliberately stricter than a sensible default so the firewall never opens
  to a range the user did not approve.
- Configurable via `/etc/rdt/rdt.conf` (host-owned; user fills in their LAN).
- Rules added with `firewall-cmd --add-rich-rule` only — no `--permanent`. They
  evaporate on reboot or firewalld reload.
- Toggle OFF probes for the rule (`--query-rich-rule`) and removes only if present.

### Decision 5: GUI = bespoke quick-settings extension

**Context**: User wanted a "simple GUI toggle, eg shell extension".
**Decision**: A minimal extension (`extension.js` + `metadata.json`, no schemas)
that adds a `QuickToggle` to the quick-settings panel. It shells out to
`~/.local/bin/rdt` for all state changes; the script remains usable on its own
from the terminal. Pattern follows `speech-to-text@fedora-desktop`.

## Risks & Mitigations

| Risk                                             | Impact | Probability | Mitigation                                                                                                                  |
| ------------------------------------------------ | ------ | ----------- | --------------------------------------------------------------------------------------------------------------------------- |
| gsettings schema path differs in GNOME 47        | Low    | Low         | Script touches the systemd user service, which is the authoritative listener. gsettings nice-to-have.                       |
| Sudo password prompt blocks extension subprocess | Med    | Low         | Repo configures passwordless sudo for primary user (per SecurityRules.md).                                                  |
| User first-toggle without configuring CIDRs      | Low    | Med         | Script aborts with explicit instructions pointing at `/etc/rdt/rdt.conf` (exit code 2). No silent partial firewall opening. |
| Extension API changes between GNOME 47/48/49     | Med    | Med         | metadata.json lists all three. QuickToggle API has been stable since GNOME 44; revisit at GNOME 50 if signature drifts.     |
