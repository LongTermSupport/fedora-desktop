## 1. Kickstart File Structure and Syntax Overview

### File Structure

A kickstart file is a plain text file with three main sections, processed in order:

```
# 1. COMMAND SECTION (kickstart commands/directives)
lang en_US.UTF-8
keyboard us
timezone America/New_York
rootpw --lock
...

# 2. PACKAGES SECTION
%packages
@core
vim
%end

# 3. SCRIPT SECTIONS (%pre, %post - can appear in any order)
%pre
...
%end

%post
...
%end
```

### Key Commands Reference (Pykickstart 3.69)

#### System Configuration

```bash
# Language and locale
lang en_US.UTF-8

# Keyboard layout
keyboard us

# Timezone (--utc means hardware clock is UTC)
timezone America/New_York --utc

# Root password (--lock disables root login, --iscrypted for hashed password)
rootpw --lock

# User creation
user --name=joseph --groups=wheel --gecos="Joseph" --password=$6$... --iscrypted

# SELinux mode
selinux --enforcing

# Firewall
firewall --enabled --ssh

# Services to enable/disable
services --enabled="sshd,NetworkManager" --disabled="kdump"
```

#### Installation Source

```bash
# Use graphical or text installer
graphical
# text

# Installation source (use closest mirror)
url --mirrorlist="https://mirrors.fedoraproject.org/mirrorlist?repo=fedora-42&arch=x86_64"

# Reboot after installation
reboot
```

#### Network Configuration

```bash
# Wired network with DHCP
network --bootproto=dhcp --device=ens3 --onboot=on --activate --hostname=fedora-workstation

# Static IP
network --bootproto=static --ip=192.168.1.100 --netmask=255.255.255.0 --gateway=192.168.1.1 --nameserver=1.1.1.1 --device=ens3 --onboot=on --activate
```

#### Bootloader

```bash
# UEFI bootloader (auto-detects EFI vs BIOS)
bootloader --location=mbr --append="rhgb quiet"

# With encrypted boot device support
bootloader --location=mbr --append="rhgb quiet" --driveorder=sda
```

#### The %include Directive

The `%include` directive allows including content from another file. This is critical for dynamic configurations generated by `%pre` scripts:

```bash
# Include dynamically generated partitioning
%include /tmp/part-include

# Include dynamically generated user config
%include /tmp/user-include
```

**GOTCHA**: Anaconda parses `%include` directives before executing `%pre` scripts. However, the actual file read happens at processing time, so files created by `%pre` are available. The `%include` directive should appear in the command section where the generated content logically belongs.

## 2. Kickstart %pre Scripts with Interactive TUI

### Overview

The `%pre` section runs before partitioning begins, in the Anaconda installer environment. It is the correct place to collect user input (LUKS passphrase, username, passwords) and generate dynamic kickstart fragments that are consumed via `%include`.

### Available Tools in the Anaconda %pre Environment

The Anaconda installation environment is a minimal Linux environment running from an initramfs/tmpfs. Available tools include:

- **Shells**: `/usr/bin/sh`, `/usr/bin/bash`
- **Python**: `/usr/libexec/platform-python` (Python 3.x)
- **Storage tools**: Commands in `/sbin` and `/bin` related to networking, storage, and file systems
- **NOT reliably available**: `dialog`, `whiptail` (these are NOT part of the standard Anaconda environment)
- **Available for I/O**: `read`, `echo`, `stty`, basic coreutils

### TTY Switching Pattern (The Reliable Method)

The standard pattern for interactive input in `%pre` is to switch to an unused TTY (typically tty6), collect input, then switch back. This is the **proven, reliable method**.

```bash
%pre --interpreter=/usr/bin/bash --log=/tmp/ks-pre.log

# Switch to tty6 for user interaction
exec < /dev/tty6 > /dev/tty6 2> /dev/tty6
chvt 6

clear
echo "========================================"
echo "  Fedora Desktop Installation Setup"
echo "========================================"
echo ""

# Collect username
echo -n "Enter your username: "
read USERNAME

# Collect full name
echo -n "Enter your full name: "
read FULLNAME

# Collect email
echo -n "Enter your email address: "
read EMAIL

# Collect LUKS passphrase (silent input with confirmation)
while true; do
    echo -n "Enter LUKS disk encryption passphrase: "
    read -s LUKSPASS
    echo ""
    echo -n "Confirm LUKS passphrase: "
    read -s LUKSPASS2
    echo ""
    if [ "$LUKSPASS" = "$LUKSPASS2" ]; then
        break
    else
        echo "Passphrases do not match. Try again."
    fi
done

# Collect user password (silent input with confirmation)
while true; do
    echo -n "Enter user password: "
    read -s USERPASS
    echo ""
    echo -n "Confirm user password: "
    read -s USERPASS2
    echo ""
    if [ "$USERPASS" = "$USERPASS2" ]; then
        break
    else
        echo "Passwords do not match. Try again."
    fi
done

echo ""
echo "Configuration collected. Proceeding with installation..."
sleep 2

# Switch back to tty1 (Anaconda's main display)
chvt 1
exec < /dev/tty1 > /dev/tty1 2> /dev/tty1

# --- Generate kickstart fragments ---

# Generate partitioning with LUKS passphrase
cat > /tmp/part-include << PARTEOF
clearpart --all --initlabel --disklabel=gpt
zerombr
part /boot/efi --fstype=efi --size=600
part /boot --fstype=ext4 --size=1024
part btrfs.01 --fstype=btrfs --size=1 --grow --encrypted --luks-version=luks2 --passphrase=${LUKSPASS} --pbkdf=pbkdf2
btrfs none --label=fedora-root btrfs.01
btrfs / --subvol --name=root LABEL=fedora-root
btrfs /home --subvol --name=home LABEL=fedora-root
PARTEOF

# Generate user configuration (hash password with python)
CRYPTPASS=$(/usr/libexec/platform-python -c "
import crypt, random, string
salt = '\$6\$' + ''.join(random.choice(string.ascii_letters + string.digits) for _ in range(16))
print(crypt.crypt('${USERPASS}', salt))
")

cat > /tmp/user-include << USEREOF
user --name=${USERNAME} --groups=wheel --gecos="${FULLNAME}" --password=${CRYPTPASS} --iscrypted
USEREOF

# Save variables for %post usage
cat > /tmp/install-vars << VARSEOF
INSTALL_USERNAME=${USERNAME}
INSTALL_FULLNAME=${FULLNAME}
INSTALL_EMAIL=${EMAIL}
VARSEOF

%end
```

### Gotchas and Limitations

1. **`dialog`/`whiptail` are NOT available** in the Anaconda installer environment by default. Using them requires injecting packages into the initramfs, which is fragile and not recommended.

2. **Keyboard issues with TUI tools**: Even if you manage to get `whiptail` into the environment, users report that Tab key inserts tab characters instead of navigating between UI elements, and arrow keys produce escape sequences. Stick with plain `read` commands.

3. **`read -s`** works reliably for password input (suppresses echo). This is the recommended way to collect passwords.

4. **`stty -echo`/`stty echo`** is an alternative to `read -s` if needed.

5. **TTY numbering**: Anaconda uses tty1 for its main display. tty6 is a safe choice for interaction. Some setups may use tty3 or tty5 depending on the Anaconda version.

6. **`%include` timing**: Place `%include /tmp/part-include` in the command section of the kickstart. Although Anaconda parses it before `%pre` runs, the actual file read happens when the command is processed, which is after `%pre` completes.

7. **Password hashing**: Use `/usr/libexec/platform-python` (available in the Anaconda environment) to hash passwords with SHA-512 via the `crypt` module. Do NOT store plaintext passwords in kickstart fragments.

8. **`/tmp` persistence**: Files written to `/tmp` in `%pre` persist and are available to both `%include` directives and `%post --nochroot` scripts (accessible at `/tmp` in the installer environment).

## 3. Kickstart LUKS + Btrfs Partitioning

### Strategy Overview

There are two main approaches:

1. **`autopart` approach** - Simple, uses Anaconda's built-in partitioning logic
2. **Manual `part` + `btrfs` approach** - Full control over partition layout, subvolumes

### Approach 1: autopart (Simple)

```bash
# Let Anaconda handle everything automatically with encryption
clearpart --all --initlabel
zerombr
autopart --type=btrfs --encrypted --passphrase=YourPassphrase --luks-version=luks2
```

**autopart --type=btrfs options:**

| Option | Description | Since |
|--------|-------------|-------|
| `--encrypted` | Encrypt all supported devices | - |
| `--passphrase=PASS` | System-wide passphrase for encrypted devices | - |
| `--luks-version=luks2` | LUKS format version | F29 |
| `--pbkdf=pbkdf2` | Key derivation function (GRUB needs pbkdf2, not argon2id) | F29 |
| `--pbkdf-memory=N` | PBKDF memory cost in KiB | F29 |
| `--pbkdf-time=N` | PBKDF time cost in ms | F29 |
| `--pbkdf-iterations=N` | Directly set PBKDF iterations | F29 |
| `--cipher=CIPHER` | Encryption cipher (e.g., aes-xts-plain64) | - |
| `--nohome` | Do not create /home partition/subvolume | - |
| `--noboot` | Do not create /boot partition | - |
| `--noswap` | Do not create swap | - |
| `--fstype=TYPE` | **Cannot be used with --type=btrfs** | - |

**CRITICAL GOTCHA**: `--type=btrfs` and `--fstype=btrfs` are NOT the same thing and cannot be combined. `--type` sets the partitioning scheme; `--fstype` sets filesystem type. With `--type=btrfs`, the filesystem is automatically btrfs.

**CRITICAL GOTCHA**: GRUB does not support Argon2id (the default LUKS2 key derivation function). If using LUKS2 with GRUB, you MUST specify `--pbkdf=pbkdf2` to ensure GRUB can unlock the encrypted volume. This is only needed if `/boot` is on the encrypted volume. If `/boot` is a separate unencrypted partition, any PBKDF works.

**Pros**: Simple, Anaconda handles EFI/boot partition creation automatically.
**Cons**: Limited control over subvolume layout, partition sizes.

### Approach 2: Manual Partitioning (Full Control)

This is the recommended approach for a production setup with specific subvolume requirements.

#### UEFI System with Separate /boot (Recommended)

```bash
# Clear all existing partitions
clearpart --all --initlabel --disklabel=gpt
zerombr

# EFI System Partition (required for UEFI)
part /boot/efi --fstype=efi --size=600

# Boot partition (unencrypted - simplifies GRUB config)
part /boot --fstype=ext4 --size=1024

# Main encrypted partition (fills remaining disk)
part btrfs.01 --fstype=btrfs --size=1 --grow \
    --encrypted --luks-version=luks2 --passphrase=YourPassphrase

# Create the Btrfs volume on the encrypted partition
btrfs none --label=fedora-root btrfs.01

# Create subvolumes
btrfs /     --subvol --name=root     LABEL=fedora-root
btrfs /home --subvol --name=home     LABEL=fedora-root
btrfs /var  --subvol --name=var      LABEL=fedora-root
```

#### Explanation of Commands

**`clearpart`**:
```bash
clearpart --all          # Remove all existing partitions
    --initlabel          # Initialize disk label (creates new partition table)
    --disklabel=gpt      # Use GPT partition table (required for UEFI)
    --drives=sda         # Optional: limit to specific drive(s)
```

**`zerombr`**: Clears the Master Boot Record. Required with `clearpart --all` for fully automated installation.

**`part`** command options for encrypted Btrfs:
```bash
part btrfs.01            # Partition ID (btrfs.XX format for Btrfs member partitions)
    --fstype=btrfs       # Filesystem type
    --size=1             # Minimum size in MiB (with --grow, this is the minimum)
    --grow               # Grow to fill available space
    --ondisk=sda         # Optional: target specific disk
    --encrypted          # Enable LUKS encryption
    --luks-version=luks2 # Use LUKS2 format
    --passphrase=PASS    # Encryption passphrase
    --pbkdf=pbkdf2       # PBKDF algorithm (only needed if /boot is encrypted too)
    --cipher=aes-xts-plain64  # Optional: specific cipher
```

**`btrfs`** command for volumes and subvolumes:
```bash
# Create a Btrfs volume (mountpoint 'none' = don't mount the volume itself)
btrfs none --label=LABEL btrfs.01 [btrfs.02 ...]
    --data=0|1|10        # Optional: RAID level for data (multi-disk only)
    --metadata=0|1|10    # Optional: RAID level for metadata (multi-disk only)

# Create a subvolume
btrfs /mountpoint --subvol --name=subvol_name LABEL=volume_label
# OR reference by volume label directly
btrfs /mountpoint --subvol --name=subvol_name volume_label
```

#### Facebook's Real-World Example (from facebookarchive/linux-desktop-kickstarts)

This is from a production kickstart used at Facebook for desktop Linux provisioning:

```bash
# Partitioning
clearpart --all --initlabel --disklabel=gpt
zerombr
bootloader --location=mbr

# EFI partition (auto-handled by reqpart in some setups)
part /boot/efi --fstype=efi --size=600

# Unencrypted Btrfs boot
part btrfs.boot --fstype=btrfs --size=1024

# Encrypted main partition with zstd compression
part btrfs.main --fstype=btrfs --size=1 --grow \
    --encrypted --passphrase=changeme \
    --fsoptions="compress=zstd:1,space_cache=v2"

# Boot volume and subvolume
btrfs /boot --label=fedora-boot btrfs.boot

# Main encrypted volume with subvolumes
btrfs none --label=fedora-btrfs btrfs.main
btrfs / --subvol --name=root LABEL=fedora-btrfs
btrfs /home --subvol --name=home LABEL=fedora-btrfs
btrfs /var/log --subvol --name=var.log LABEL=fedora-btrfs
btrfs /var/lib/libvirt/images --subvol --name=var.lib.libvirt.images LABEL=fedora-btrfs
```

**Key observations from Facebook's example:**
- They use a Btrfs boot partition (not ext4) - this works but ext4 is simpler
- The `--fsoptions` parameter passes mount options like zstd compression directly
- Subvolume names use dots instead of slashes (e.g., `var.log` not `var/log`)
- `/boot` is a separate Btrfs volume (unencrypted), not on the LUKS partition

#### The `reqpart` Alternative for Boot Partitions

Instead of manually specifying EFI and boot partitions, you can use `reqpart`:

```bash
# Automatically create required platform partitions
reqpart --add-boot

# This creates:
# - /boot/efi on UEFI systems (EFI System Partition)
# - biosboot on BIOS+GPT systems
# - /boot partition (when --add-boot is specified)
```

**GOTCHA**: `reqpart` cannot be used with `autopart`. It is designed for use with manual `part` commands for the remaining disk space.

### LUKS2 and GRUB Compatibility

**The key issue**: GRUB 2.06+ supports LUKS2, BUT only with PBKDF2 as the key derivation function. LUKS2 defaults to Argon2id, which GRUB cannot handle.

**Solutions (in order of preference for our use case):**

1. **Separate unencrypted /boot** (Recommended): Keep `/boot` on an unencrypted partition. GRUB only needs to read `/boot`, so LUKS passphrase prompt happens via the initramfs, not GRUB. Any PBKDF works on the encrypted partition.

2. **LUKS2 with PBKDF2**: If you want encrypted `/boot`, specify `--pbkdf=pbkdf2` on the encrypted partition. This is less secure than Argon2id but GRUB-compatible.

3. **LUKS1 for /boot, LUKS2 for root**: Complex setup, not recommended for kickstart automation.

## 4. Kickstart %post Scripts

### Overview

The `%post` section runs after the system has been installed and packages have been applied. It is the primary place for post-installation customization. You can have **multiple** `%post` sections, each independent.

### Chroot vs Non-Chroot

#### Default (chroot into installed system)

```bash
%post --interpreter=/usr/bin/bash --log=/root/ks-post.log
# Runs inside the newly installed system (chrooted)
# /root is the installed system's /root
# Can use dnf, systemctl (with caveats), modify config files
dnf install -y git ansible
echo "Post-install completed" >> /root/ks-post.log
%end
```

#### Non-chroot (installer environment)

```bash
%post --nochroot --log=/tmp/ks-post-nochroot.log
# Runs in the installer environment
# The installed system is mounted at /mnt/sysimage
# Useful for copying files from installer media to installed system
cp /tmp/install-vars /mnt/sysimage/root/install-vars
%end
```

### %post Options

| Option | Description |
|--------|-------------|
| `--interpreter=/path` | Script interpreter (default: `/bin/sh`) |
| `--log=/path/file` | Log output to file (path relative to chroot) |
| `--nochroot` | Run in installer environment, not chroot |
| `--erroronfail` | Halt installation if script fails |
| `--logfile=/path` | Alias for --log |

### Network Access in %post

**This is one of the most important considerations.**

#### With Static IP Configuration
If `network --bootproto=static` was used with a nameserver, full network access (including DNS resolution) is available in `%post`.

#### With DHCP
**GOTCHA**: When using DHCP, `/etc/resolv.conf` may NOT be populated when `%post` runs. You can access the network by IP address, but DNS resolution may fail.

**Workarounds for DHCP:**

```bash
%post --interpreter=/usr/bin/bash --log=/root/ks-post.log

# Workaround 1: Force DHCP renewal
dhclient

# Workaround 2: Manually set DNS
echo "nameserver 1.1.1.1" > /etc/resolv.conf
echo "nameserver 8.8.8.8" >> /etc/resolv.conf

# Workaround 3: Copy from installer environment (use separate --nochroot section)
# In %post --nochroot:
# cp /etc/resolv.conf /mnt/sysimage/etc/resolv.conf

# Now network operations should work
dnf install -y git ansible-core
%end
```

### Can %post Clone Git Repos and Run Ansible?

**YES**, with caveats:

```bash
%post --interpreter=/usr/bin/bash --log=/root/ks-post.log --erroronfail

# Ensure DNS works
echo "nameserver 1.1.1.1" > /etc/resolv.conf

# Install required packages
dnf install -y git ansible-core python3-libdnf5

# Clone the repository
git clone https://github.com/username/fedora-desktop.git /root/fedora-desktop

# Run ansible-playbook (within the chroot)
cd /root/fedora-desktop
ansible-playbook playbooks/playbook-main.yml --connection=local

%end
```

**Limitations of running Ansible in %post:**

1. **systemctl commands fail in chroot**: Most `systemctl` commands refuse to work because systemd in the chroot is not the running PID 1. Services can be enabled (symlink-based) but not started.

2. **No running services**: NetworkManager, sshd, etc. are not running in the chroot. Anything that depends on a running service will fail.

3. **Limited hardware access**: Some hardware configuration that requires running services or kernel modules may not work.

4. **User sessions don't exist**: GNOME settings, dconf, and other user-session-dependent configurations won't work.

5. **Timeouts**: Long-running operations in %post can cause Anaconda to appear hung. There's no progress indication.

### The First-Boot Service Pattern (Recommended for Ansible)

Instead of running the full Ansible setup in %post, create a systemd service that runs on first boot:

```bash
%post --interpreter=/usr/bin/bash --log=/root/ks-post.log --erroronfail

# Ensure DNS works
echo "nameserver 1.1.1.1" > /etc/resolv.conf

# Install minimal dependencies
dnf install -y git ansible-core python3-libdnf5

# Clone the configuration repository
git clone https://github.com/username/fedora-desktop.git /root/fedora-desktop

# Create a first-boot setup script
cat > /usr/local/bin/firstboot-setup.sh << 'FBEOF'
#!/bin/bash
set -euo pipefail
exec > /var/log/firstboot-setup.log 2>&1

echo "Starting first-boot setup at $(date)"

# Run the Ansible playbook
cd /root/fedora-desktop
ansible-playbook playbooks/playbook-main.yml --connection=local

# Disable the service after successful run
systemctl disable firstboot-setup.service

echo "First-boot setup completed at $(date)"
FBEOF
chmod +x /usr/local/bin/firstboot-setup.sh

# Create systemd service
cat > /etc/systemd/system/firstboot-setup.service << 'SVCEOF'
[Unit]
Description=First Boot System Configuration (Ansible)
After=network-online.target
Wants=network-online.target
ConditionPathExists=/root/fedora-desktop

[Service]
Type=oneshot
ExecStart=/usr/local/bin/firstboot-setup.sh
RemainAfterExit=yes
TimeoutStartSec=3600

[Install]
WantedBy=multi-user.target
SVCEOF

# Enable the service
systemctl enable firstboot-setup.service

%end
```

### Passing Variables from %pre to %post

Since `%pre` runs in the installer environment and `%post` (default) runs in a chroot, you need to bridge the gap:

```bash
# In %pre: write to /tmp/install-vars
cat > /tmp/install-vars << EOF
USERNAME=joseph
EMAIL=joseph@example.com
EOF

# In %post --nochroot: copy to installed system
%post --nochroot
cp /tmp/install-vars /mnt/sysimage/tmp/install-vars
%end

# In %post (chroot): read the variables
%post
source /tmp/install-vars
echo "Setting up for user: $USERNAME"
%end
```

## 5. Kickstart + WiFi

### Current State of WiFi Support in Kickstart

WiFi support in Kickstart/Anaconda is **limited and somewhat unreliable**. The situation has evolved over different Fedora/RHEL versions.

### The `network` Command WiFi Options

```bash
# Pykickstart network command with wireless options
network --device=wlan0 --essid="MyNetwork" --wpakey="MyPassword" --activate --onboot=on
```

**Available wireless-specific options:**

| Option | Description | Status |
|--------|-------------|--------|
| `--essid=SSID` | Wireless network SSID | Added F3, potentially deprecated |
| `--wpakey=KEY` | WPA/WPA2 passphrase | Added later, potentially deprecated |
| `--wepkey=KEY` | WEP key (obsolete security) | Deprecated |

### Boot Parameter WiFi Options

WiFi can also be specified via Anaconda boot parameters:

```
inst.essid=MyNetwork inst.wpakey=MyPassword
```

**WARNING**: These boot parameters (`essid`, `wpakey`) were listed as **removed** in newer Anaconda documentation. Their availability depends on your specific Fedora version.

### Practical WiFi Strategies

#### Strategy 1: Pre-connect WiFi in Anaconda GUI, then apply kickstart
Boot the installer, connect to WiFi through the Anaconda GUI, then apply the kickstart from a network location. This is semi-automated but reliable.

#### Strategy 2: Use %pre to configure WiFi via nmcli

```bash
%pre --interpreter=/usr/bin/bash
# Attempt to connect to WiFi using nmcli
nmcli device wifi connect "MySSID" password "MyPassword" ifname wlan0
sleep 10  # Wait for connection
%end
```

**GOTCHA**: This may or may not work depending on whether NetworkManager and WiFi drivers are loaded in the Anaconda environment.

#### Strategy 3: Use a wired connection for installation
The most reliable approach. Use Ethernet for the kickstart installation, then configure WiFi in `%post` for post-installation use:

```bash
%post
# Configure WiFi for post-installation use
nmcli connection add type wifi con-name "HomeWiFi" ifname wlan0 ssid "MySSID"
nmcli connection modify "HomeWiFi" wifi-sec.key-mgmt wpa-psk wifi-sec.psk "MyPassword"
nmcli connection modify "HomeWiFi" connection.autoconnect yes
%end
```

#### Strategy 4: USB kickstart with local packages
Put the kickstart file and all needed packages on a USB drive. No network needed during install. Configure WiFi in %post for first-boot use.

### Recommendation

**For our use case**: Use Ethernet for the installation phase. Configure WiFi in `%post` if needed. WiFi during Anaconda kickstart installation is unreliable and not worth the complexity.

## 6. Kickstart Serving Methods

### Method 1: Embedded in ISO (mkksiso) -- RECOMMENDED

The `mkksiso` tool from the `lorax` package embeds a kickstart file directly into the Fedora installation ISO.

```bash
# Install mkksiso
sudo dnf install lorax

# Basic usage (must be run as root for full UEFI support)
sudo mkksiso --ks /path/to/ks.cfg /path/to/Fedora-Workstation-Live-x86_64-42.iso /path/to/Fedora-Custom.iso

# With custom volume ID
sudo mkksiso --ks /path/to/ks.cfg -V "Fedora-42-Custom" /path/to/Fedora-42.iso /path/to/Fedora-42-Custom.iso

# With additional kernel command line arguments
sudo mkksiso --ks /path/to/ks.cfg --cmdline "inst.text" /path/to/Fedora-42.iso /path/to/Fedora-42-Custom.iso

# Add a local repo directory to the ISO
sudo mkksiso --ks /path/to/ks.cfg --add /path/to/repo/ /path/to/Fedora-42.iso /path/to/Fedora-42-Custom.iso

# Without UEFI (can run as non-root)
mkksiso --ks /path/to/ks.cfg --skip-mkefiboot /path/to/Fedora-42.iso /path/to/Fedora-42-Custom.iso
```

**What mkksiso does:**
- Adds the kickstart file to the root of the ISO
- Automatically appends `inst.ks=...` to the kernel command line in the ISO boot configuration
- Preserves EFI and Mac boot support from the original ISO
- Can add additional files and repos to the ISO

**Pros:**
- Self-contained, no network dependency during boot
- Works offline
- Simple to distribute (just flash to USB)
- Kickstart is automatically applied at boot

**Cons:**
- Must rebuild ISO when kickstart changes
- Requires root for UEFI-bootable ISOs (as of lorax 38.4+)
- ISO size increases with added repos

### Method 2: HTTP/HTTPS Server

```bash
# Serve from any HTTP server
# Boot parameter:
inst.ks=https://example.com/ks/fedora-desktop.ks

# From GitHub raw URL (convenient but security risk)
inst.ks=https://raw.githubusercontent.com/user/repo/branch/ks.cfg
```

**Pros:**
- Easy to update (edit file on server)
- Can version control the kickstart file
- Works with PXE/network boot

**Cons:**
- Requires network access during boot
- Security risk: kickstart is fetched over network (use HTTPS)
- GitHub raw URLs can change or become rate-limited
- Passwords/secrets in the kickstart file are transmitted over network

### Method 3: USB Drive (Side-by-side)

Place the kickstart file on a FAT-formatted USB drive alongside or separate from the installation media.

```bash
# Boot parameter using filesystem label
inst.ks=hd:LABEL=KSDATA:/ks.cfg

# Using device path
inst.ks=hd:sdb1:/ks.cfg

# Using UUID
inst.ks=hd:UUID=xxxx-xxxx:/ks.cfg
```

**Pros:**
- No network needed
- Easy to update (edit file on USB)
- Can carry additional files (repos, keys)

**Cons:**
- Requires a separate USB drive or partition
- Must manually edit boot parameters
- Device naming (sdb1) can be unpredictable

### Method 4: NFS Share

```bash
inst.ks=nfs:server.example.com:/exports/ks/fedora-desktop.ks
# With NFS options
inst.ks=nfs:nfsvers=4:server.example.com:/exports/ks/fedora-desktop.ks
```

**Pros:**
- Good for enterprise/lab environments
- Central management of kickstart files

**Cons:**
- Requires NFS infrastructure
- Network dependency
- Overkill for personal/small deployments

### Method 5: FTP

```bash
inst.ks=ftp://server.example.com/ks/fedora-desktop.ks
```

Not recommended -- legacy protocol, unencrypted, prefer HTTPS.

### Recommendation for This Project

**Primary method**: `mkksiso` to embed the kickstart into a custom Fedora ISO. This gives a single, self-contained USB stick that boots directly into the automated installation.

**Secondary method**: Kickstart on a separate FAT-formatted USB partition, referenced via `inst.ks=hd:LABEL=...`. This allows quick iteration during development without rebuilding the ISO.

## 8. inst.ks Boot Parameter

### Complete Syntax Reference

The `inst.ks=` boot parameter tells Anaconda where to find the kickstart file. It is appended to the kernel command line.

#### Syntax Formats

```bash
# HTTP/HTTPS (most common for network installs)
inst.ks=http://server.example.com/path/ks.cfg
inst.ks=https://server.example.com/path/ks.cfg

# FTP
inst.ks=ftp://server.example.com/path/ks.cfg

# NFS
inst.ks=nfs:server.example.com:/path/ks.cfg
inst.ks=nfs:nfsvers=4:server.example.com:/path/ks.cfg

# Local hard drive or USB (by device name)
inst.ks=hd:sdb1:/ks.cfg

# Local hard drive or USB (by filesystem LABEL - recommended)
inst.ks=hd:LABEL=MyUSB:/ks.cfg

# Local hard drive or USB (by UUID)
inst.ks=hd:UUID=1234-ABCD:/ks.cfg

# Already on the initrd (embedded via mkksiso)
# No inst.ks needed - mkksiso adds it automatically

# File on the installation media itself
inst.ks=cdrom:/ks.cfg
```

### How to Apply the Boot Parameter

#### Method 1: Edit GRUB Menu at Boot

1. Boot from Fedora installation media
2. At the GRUB boot menu, press `e` to edit the default entry
3. Find the line starting with `linuxefi` (UEFI) or `linux` (BIOS)
4. Append `inst.ks=<location>` to the end of that line
5. Press `Ctrl+X` or `F10` to boot with the modified parameters

Example of the edited line:
```
linuxefi /images/pxeboot/vmlinuz inst.stage2=hd:LABEL=Fedora-WS-Live-x86_64-42 quiet inst.ks=hd:LABEL=KSDATA:/ks.cfg
```

#### Method 2: Modify ISO Boot Configuration (permanent)

For a permanent modification, edit the ISO's GRUB configuration before creating the bootable media. The `mkksiso` tool does this automatically.

### Additional Useful Boot Parameters

```bash
# Force text mode installation
inst.text

# Force graphical mode
inst.graphical

# Specify installation source

# Network configuration at boot
```

## 9. Production Kickstart Strategy Recommendations

Based on all the research above, here is the recommended approach for the fedora-desktop project.

### Recommended Architecture

```
kickstart/
  ks.cfg                    # Main kickstart file
  README.md                 # Usage instructions
  build-iso.sh              # Script to embed ks.cfg into Fedora ISO using mkksiso
```

### Recommended Kickstart Design

#### Overall Flow

```
1. %pre script:
   - Switch to tty6
   - Collect: LUKS passphrase, username, full name, email
   - Generate /tmp/part-include (partitioning with LUKS passphrase)
   - Generate /tmp/user-include (user creation with hashed password)
   - Generate /tmp/install-vars (variables for %post)
   - Switch back to tty1

2. Command section:
   - Basic system config (lang, keyboard, timezone, etc.)
   - %include /tmp/part-include (dynamic partitioning)
   - %include /tmp/user-include (dynamic user creation)
   - Network config (DHCP wired)
   - Bootloader config

3. %packages section:
   - @^workstation-product-environment
   - Additional packages needed by Ansible (git, ansible-core, python3-libdnf5)
   - Exclusions for unwanted packages

4. %post --nochroot:
   - Copy /tmp/install-vars to /mnt/sysimage/root/

5. %post (chroot):
   - Fix DNS if needed
   - Clone fedora-desktop git repo
   - Create firstboot systemd service for Ansible
   - Basic system configuration that doesn't need running services

6. First boot:
   - Systemd service runs Ansible playbook
   - Full system configuration applied
   - Service disables itself after completion
```

#### Why Not Run Ansible Directly in %post?

1. **systemctl limitations**: Most service management fails in chroot
2. **No user session**: GNOME/dconf settings require a running user session
3. **No display server**: GUI-dependent configurations fail
4. **Timeout risk**: Long Ansible runs may cause Anaconda issues
5. **Debugging difficulty**: Failures in %post are harder to diagnose
6. **First-boot gives full environment**: All services running, network up, user logged in

#### Security Considerations

1. **LUKS passphrase**: Collected interactively in %pre, written to /tmp (tmpfs, gone after reboot). Never stored permanently in plaintext.

2. **User password**: Hashed with SHA-512 before being written to kickstart fragment. Plaintext only exists momentarily in bash variables.

3. **Kickstart file itself**: Should NOT contain plaintext passwords. All secrets should be collected via %pre or injected at build time.

4. **mkksiso ISO**: The embedded kickstart is readable if someone mounts the ISO. Since secrets are collected at install time (not embedded), this is acceptable.

### Partitioning Recommendation

```bash
# For UEFI system with LUKS2 + Btrfs
clearpart --all --initlabel --disklabel=gpt
zerombr
bootloader --location=mbr --append="rhgb quiet"

# 600 MiB EFI System Partition
part /boot/efi --fstype=efi --size=600

# 1 GiB ext4 boot (unencrypted for GRUB simplicity)
part /boot --fstype=ext4 --size=1024

# Remaining space: LUKS2-encrypted Btrfs
part btrfs.01 --fstype=btrfs --size=1 --grow \
    --encrypted --luks-version=luks2 --passphrase=${LUKSPASS}

# Btrfs volume
btrfs none --label=fedora btrfs.01

# Subvolumes (matching Fedora defaults)
btrfs /     --subvol --name=root LABEL=fedora
btrfs /home --subvol --name=home LABEL=fedora
```

This keeps `/boot` unencrypted (separate ext4 partition), which means:
- GRUB does not need to unlock LUKS (simpler, more reliable)
- Any PBKDF algorithm works (can use Argon2id, the LUKS2 default)
- LUKS passphrase prompt happens via Plymouth/initramfs (better UX)
- Single passphrase entry at boot

### Skeleton Kickstart File

```bash
#version=F42

# ============================================================
# PRE-INSTALLATION: Collect user input
# ============================================================
%pre --interpreter=/usr/bin/bash --log=/tmp/ks-pre.log
exec < /dev/tty6 > /dev/tty6 2> /dev/tty6
chvt 6

clear
echo "========================================"
echo "  Fedora 42 Desktop - Automated Setup"
echo "========================================"
echo ""

read -p "Username: " KS_USERNAME
read -p "Full name: " KS_FULLNAME
read -p "Email: " KS_EMAIL

while true; do
    read -s -p "LUKS passphrase: " KS_LUKS1; echo
    read -s -p "Confirm passphrase: " KS_LUKS2; echo
    [ "$KS_LUKS1" = "$KS_LUKS2" ] && break
    echo "Mismatch. Try again."
done

while true; do
    read -s -p "User password: " KS_PASS1; echo
    read -s -p "Confirm password: " KS_PASS2; echo
    [ "$KS_PASS1" = "$KS_PASS2" ] && break
    echo "Mismatch. Try again."
done

echo "Proceeding with installation..."
sleep 2
chvt 1
exec < /dev/tty1 > /dev/tty1 2> /dev/tty1

# Hash password
KS_CRYPTPASS=$(/usr/libexec/platform-python -c "
import crypt, random, string
salt = '\$6\$' + ''.join(random.choice(string.ascii_letters + string.digits) for _ in range(16))
print(crypt.crypt('${KS_PASS1}', salt))
")

# Generate partitioning
cat > /tmp/part-include << EOF
clearpart --all --initlabel --disklabel=gpt
zerombr
part /boot/efi --fstype=efi --size=600
part /boot --fstype=ext4 --size=1024
part btrfs.01 --fstype=btrfs --size=1 --grow --encrypted --luks-version=luks2 --passphrase=${KS_LUKS1}
btrfs none --label=fedora btrfs.01
btrfs / --subvol --name=root LABEL=fedora
btrfs /home --subvol --name=home LABEL=fedora
EOF

# Generate user config
cat > /tmp/user-include << EOF
user --name=${KS_USERNAME} --groups=wheel --gecos="${KS_FULLNAME}" --password=${KS_CRYPTPASS} --iscrypted
EOF

# Save vars for %post
cat > /tmp/install-vars << EOF
KS_USERNAME=${KS_USERNAME}
KS_FULLNAME=${KS_FULLNAME}
KS_EMAIL=${KS_EMAIL}
EOF

%end

# ============================================================
# SYSTEM CONFIGURATION
# ============================================================
lang en_US.UTF-8
keyboard us
timezone America/New_York --utc
rootpw --lock
selinux --enforcing
firewall --enabled --ssh
network --bootproto=dhcp --onboot=on --activate

# Bootloader
bootloader --location=mbr --append="rhgb quiet"

# Dynamic includes from %pre
%include /tmp/part-include
%include /tmp/user-include

# Reboot after install
reboot

# ============================================================
# PACKAGES
# ============================================================
%packages --excludeWeakdeps
@^workstation-product-environment
vim-enhanced
git
gh
ansible-core
python3-libdnf5
htop
wget
bash-completion
ripgrep
jq
openssl
-gnome-boxes
-gnome-tour
%end

# ============================================================
# POST: Copy install vars (nochroot)
# ============================================================
%post --nochroot --log=/tmp/ks-post-nochroot.log
cp /tmp/install-vars /mnt/sysimage/root/install-vars
%end

# ============================================================
# POST: System configuration (chroot)
# ============================================================
%post --interpreter=/usr/bin/bash --log=/root/ks-post.log --erroronfail

# Ensure DNS works
echo "nameserver 1.1.1.1" >> /etc/resolv.conf

# Source install variables
source /root/install-vars

# Clone the fedora-desktop repo
git clone https://github.com/${KS_USERNAME}/fedora-desktop.git /root/fedora-desktop || true

# Create first-boot Ansible service
cat > /usr/local/bin/fedora-desktop-setup.sh << 'SETUPEOF'
#!/bin/bash
set -euo pipefail
exec >> /var/log/fedora-desktop-setup.log 2>&1
echo "=== First-boot setup started at $(date) ==="
source /root/install-vars
cd /root/fedora-desktop
# Run the Ansible playbook
ansible-playbook playbooks/playbook-main.yml --connection=local
systemctl disable fedora-desktop-setup.service
echo "=== First-boot setup completed at $(date) ==="
SETUPEOF
chmod +x /usr/local/bin/fedora-desktop-setup.sh

cat > /etc/systemd/system/fedora-desktop-setup.service << 'SVCEOF'
[Unit]
Description=Fedora Desktop First-Boot Configuration (Ansible)
After=network-online.target
Wants=network-online.target
ConditionPathExists=/root/fedora-desktop

[Service]
Type=oneshot
ExecStart=/usr/local/bin/fedora-desktop-setup.sh
RemainAfterExit=yes
TimeoutStartSec=3600

[Install]
WantedBy=multi-user.target
SVCEOF

systemctl enable fedora-desktop-setup.service

# Clean up sensitive data
rm -f /root/install-vars

%end
```

### Build and Deploy Script

```bash
#!/bin/bash
# build-iso.sh - Create a custom Fedora installation ISO with embedded kickstart
set -euo pipefail

FEDORA_ISO="${1:?Usage: $0 <fedora-iso> [output-iso]}"
OUTPUT_ISO="${2:-fedora-desktop-custom.iso}"
KS_FILE="$(dirname "$0")/ks.cfg"

if [ ! -f "$KS_FILE" ]; then
    echo "Error: Kickstart file not found at $KS_FILE"
    exit 1
fi

echo "Building custom ISO..."
echo "  Source: $FEDORA_ISO"
echo "  Kickstart: $KS_FILE"
echo "  Output: $OUTPUT_ISO"

sudo mkksiso --ks "$KS_FILE" "$FEDORA_ISO" "$OUTPUT_ISO"

echo "Done! Flash to USB with:"
echo "  sudo dd if=$OUTPUT_ISO of=/dev/sdX bs=4M status=progress oflag=sync"
```

## Appendix A: Common Gotchas Summary

| Gotcha | Impact | Solution |
|--------|--------|----------|
| `--type=btrfs` vs `--fstype=btrfs` | Cannot combine both in autopart | Use `--type=btrfs` only |
| GRUB + LUKS2 + Argon2id | GRUB cannot unlock partition | Use separate /boot or `--pbkdf=pbkdf2` |
| DHCP + DNS in %post | DNS resolution fails | Add `nameserver` to `/etc/resolv.conf` |
| systemctl in %post chroot | Commands refuse to run | Use firstboot service pattern |
| `dialog`/`whiptail` in %pre | Not available in Anaconda | Use plain `read` with TTY switching |
| %include before %pre | File doesn't exist yet | Anaconda handles this - file read is deferred |
| WiFi in kickstart | Unreliable/deprecated options | Use Ethernet for install |
| Plaintext LUKS passphrase | Security risk if stored in file | Collect in %pre, only in tmpfs |
| %post --nochroot paths | Installed system at /mnt/sysimage | Use full path: `/mnt/sysimage/root/...` |
| mkksiso needs root | UEFI boot support requires root | Run with sudo |
| Multiple encrypted partitions | Each needs its own passphrase prompt | Use single encrypted partition |
