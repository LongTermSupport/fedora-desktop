# Plan 022: Install Security and Resilience

**Status**: 🔄 In Progress
**Created**: 2026-03-09
**Owner**: Agent
**Priority**: High

## Overview

Two problems discovered during real-hardware testing of the Fedora kickstart installer:

1. **FDINST partition gets wiped on reinstall** — `clearpart --all` in the generated
   `/tmp/part-include` destroys everything, including the FDINST partition that holds
   the netinstall ISO and squashfs.img. After a reinstall you must re-run
   `setup-netinstall-boot.bash` (downloading ~3GB again) before the install entry is
   usable again. This is slow and avoidable.

2. **Anyone can trigger a reinstall from the GRUB menu** — the install entry has no
   authentication. A curious person or accidental keypress wipes the laptop. We need
   both a GRUB-level gate and a kickstart `%pre` gate to prevent this.

**Bonus issue raised during testing**: The GRUB menu font is too small on HiDPI/laptop
displays. The install entry already sets a large console font (`latarcyrheb-sun32`) via
the kernel command line — the same fix should apply to the GRUB menu itself.

**Related known issue** (not the primary focus but tracked here): After the first
successful install the user could not log in — user password was reported wrong. Root
cause uncertain; likely a keyboard layout mismatch during `%pre` password entry, or
the `openssl passwd -6` invocation behaving unexpectedly. This is noted for follow-up
investigation (see Phase 4).

## Goals

- FDINST partition survives reinstalls: only EFI/boot/LUKS are wiped, FDINST is preserved
- A GRUB password prevents the install entry from being selected without authentication
- A kickstart `%pre` authorization check provides a second layer before any disk is touched
- GRUB menu font is legible on HiDPI displays
- `./scripts/qa-all.bash` passes after all changes

## Non-Goals

- UEFI/BIOS supervisor password (out of scope — requires firmware configuration)
- Supporting multiple protected users in GRUB (one install password is sufficient)
- Redesigning the partition layout (sizes, subvolumes, etc. are unchanged)

## Context & Background

### FDINST Preservation Research

Kickstart's `clearpart --list=sda1,sda2,sda3` clears only the named partitions, leaving
anything not listed (e.g., `sda4` = FDINST) untouched. This was broken in Fedora 25/28
(BZ#1561930) but fixed in `anaconda-21.48.22.136-1` (RHBA-2018:3035). Fedora 43 is
not affected.

**Two-case logic required in `%pre`:**

- **Reinstall** (FDINST exists by label `FDINST`): use `clearpart --list` naming only
  the three OS partitions; detect disk + partition names dynamically via `lsblk`.
- **First install** (no FDINST): keep `clearpart --all --initlabel --disklabel=gpt` +
  `zerombr` exactly as today.

`zerombr` and `--initlabel` must NOT be used in the reinstall case — they would
reinitialise the GPT and destroy FDINST.

The generated `part` directives for the reinstall case must include `--ondisk=<disk>`
to constrain Anaconda to the correct disk, since FDINST's presence removes the ambiguity.
The FDINST partition needs no `part` directive — Anaconda simply ignores partitions not
referenced.

### GRUB Password Research

GRUB2 authentication is controlled by `set superusers` and `password_pbkdf2`:

```grub
set superusers="grub-install"
password_pbkdf2 grub-install grub.pbkdf2.sha512.10000.<hash>

menuentry "Fedora 43 Install (ISO)" --users "" {
    ...
}
```

`--users ""` on the install entry means superusers only. On Fedora 43 with BLS,
normal kernel entries in `/boot/loader/entries/*.conf` already carry
`grub_arg --unrestricted` — they remain accessible without any password even after
`set superusers` is active. No changes to `10_linux` or BLS files are needed.

The hash is generated by:
```bash
printf '%s\n%s\n' "$password" "$password" | grub2-mkpasswd-pbkdf2 2>&1 \
    | grep -oP 'grub\.pbkdf2\.\S+'
```

The superuser definition and protected entry are both placed in
`/etc/grub.d/40_fedora_netinstall` (the file `setup-netinstall-boot.bash` already
manages). No separate file needed.

### Kickstart `%pre` Authorization Layer

The `%pre` section already runs on tty6 with the keyboard attached. Adding a short
password prompt at the very beginning (before WiFi, before FDINST mount, before any
disk operations) gives a second gate. If wrong: clear screen, show error, restore
tty1, exit — Anaconda aborts with no damage done.

The authorization code is embedded in the ks.cfg copy on `/boot` (not the repo
source). `setup-netinstall-boot.bash` prompts for it during setup and `sed`-substitutes
its hash into the deployed copy, exactly as it already does for keyboard layout,
branch name, and squashfs filename.

The secret is stored as a hash in the ks.cfg on the unencrypted `/boot` partition.
Root on the running system could read it. This is acceptable: the threat model is
_accidental or low-sophistication_ reinstall, not a root-privileged attacker.
The two secrets (GRUB password and `%pre` auth code) should be different.

### GRUB Menu Font Size

`setup-netinstall-boot.bash` already calls `grub2-editenv - unset menu_auto_hide` and
`grub2-mkconfig`. The GRUB menu font can be set by placing a `GRUB_FONT` variable in
`/etc/default/grub` (or emitting a `set gfxmode` / `loadfont` block in a grub.d
script). The standard approach: generate a GRUB-compatible `.pf2` font from a system
font at a larger size using `grub2-mkfont`, then reference it in the GRUB config.

A simpler reliable approach: add `GRUB_GFXMODE=auto` and emit a `set gfxfont` in a
`00_header`-sourced snippet. Fedora ships `unicode.pf2` at `/boot/grub2/fonts/` by
default; we can generate a larger version (`unicode-large.pf2`) during setup.

## Technical Decisions

### Decision 1: `clearpart --list` vs `clearpart --none` + `--onpart`

**Options:**
1. `clearpart --list=p1,p2,p3` — wipe only named partitions, create new ones with `--ondisk`
2. `clearpart --none` + `part --onpart=p1` for each — wipe nothing, reuse exact slots

**Decision**: Use `clearpart --list` (Option 1). It is simpler (no need to know existing
sizes), cleaner (Anaconda manages new partition creation), and the old bug is fixed in
Fedora 43. Using `--ondisk` rather than `--onpart` lets Anaconda create appropriately
sized partitions without needing to know what was there before.

### Decision 2: GRUB password scope — one file vs two files

**Options:**
1. Embed `set superusers` + `password_pbkdf2` + protected `menuentry` all in
   `40_fedora_netinstall`
2. Separate `01_users` for credentials + `40_fedora_netinstall` for entry

**Decision**: Option 1 (single file). Simpler, self-contained. The `--remove` path
already deletes `40_fedora_netinstall`, which cleanly removes both the credentials and
the entry in one operation. A separate `01_users` would need separate cleanup logic.

### Decision 3: `%pre` auth code storage — plaintext vs hash

**Decision**: Store as an `openssl passwd -6` hash in the deployed ks.cfg. Plaintext
would be trivially readable from `/boot`. The hash is brute-forceable from `/boot` if
someone has root, but that's outside the threat model. The SHA-512 hash is sufficient
protection against casual inspection.

### Decision 4: GRUB font approach

**Options:**
1. Generate `.pf2` from system font with `grub2-mkfont` during setup
2. Set `GRUB_FONT` in `/etc/default/grub` to reference the generated file
3. Emit `loadfont` in the grub.d script directly

**Decision**: Generate the font (Option 1) + reference via `GRUB_FONT` in
`/etc/default/grub` (Option 2). The `GRUB_FONT` variable is the official Fedora
mechanism and survives `grub2-mkconfig` regenerations. Soft-fail if font generation
fails (the font is cosmetic, not functional).

## Risks & Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| `clearpart --list` with partition names fails in Anaconda 43 | High — install fails | Low — bug fixed since F28 | Test with ksvalidator first; fallback to `clearpart --none` + `--onpart` |
| `lsblk` not available in initrd for `%pre` | High — can't detect partitions | Low — lsblk ships in installer | Use `blkid -o device -t TYPE=vfat` as fallback to enumerate partitions |
| GRUB password locks out normal boot | Critical | Low — BLS entries have `--unrestricted` | Verify BLS files before and after: `grep grub_arg /boot/loader/entries/*.conf` |
| `grub2-mkpasswd-pbkdf2` not available | Medium — GRUB entry created without password | Low — ships with grub2-tools | Soft-fail: create unprotected entry + print loud warning |
| Font generation fails (font file path varies) | Low — cosmetic only | Medium — path varies by Fedora version | Try multiple font paths, soft-fail |
| User password issue (login failure) reproduces | High — laptop locked out again | Medium — root cause unknown | Phase 5 investigation + test on VM before hardware |
